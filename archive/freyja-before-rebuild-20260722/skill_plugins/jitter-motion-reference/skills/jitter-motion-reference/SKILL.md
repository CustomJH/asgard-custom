# Jitter Motion Reference

Jitter 템플릿을 복사본이 아니라 **동작 레퍼런스**로 사용하라. 프로젝트의 토큰·컴포넌트·브랜드를 유지하면서 관찰한 모션 문법만 재구현하라.

## 적용 위계

1. 프로젝트 디자인 시스템과 사용자 요구를 먼저 지켜라.
2. `asgard-freyja-motion`의 접근성·성능·타이밍 계약을 지켜라. 이 스킬은 예제 선택을 보강할 뿐 수치 캐논을 대체하지 않는다.
3. 영상 산출물이면 `asgard-freyja-video`, 차트면 해당 데이터 애니메이션 스킬을 함께 사용하라.
4. Jitter 예제의 로고·이미지·브랜드 자산은 복제하거나 재배포하지 말라. 이름과 링크를 출처로 남기고 모션 구조만 새 자산으로 재해석하라.

## 작업 흐름

1. **표면을 분류하라.** UI 피드백, 화면 전환, 타이포, 로고·아이콘, 데이터, 배경·효과, 쇼릴·소셜, 공간·3D 중 하나를 주축으로 고르라.
2. **레퍼런스를 좁혀라.** [패턴 아틀라스](references/pattern-atlas.md)에서 주 레퍼런스 1개와 필요할 때만 보조 1개를 선택하라. 최신성이 중요하면 원문 카탈로그를 다시 확인하라.
3. **외형이 아니라 문법을 추출하라.** 다음을 명시하라: 목적, 시작·종료 상태, 변형 축, 레이어 순서, 타이밍·이징, 반복 조건, 사용자 입력과의 관계.
4. **현재 스택으로 번역하라.** 기존 CSS·Web Animations API·프로젝트 내 모션 의존성을 먼저 사용하라. Jitter 파일이나 새 라이브러리를 기본 전제로 두지 말라.
5. **정지 상태도 완성하라.** 애니메이션이 없거나 끝난 상태에서도 정보 구조와 조작 가능성이 유지되게 하라.
6. **실제 결과를 확인하라.** 진입·중간·완료 상태를 캡처하고 콘솔 오류, 레이아웃 이동, 프레임 드롭, 키보드·터치 상태, `prefers-reduced-motion` 대안을 검사하라.

## 모션 브리프 형식

구현 전 또는 결과 보고에 아래를 짧게 남겨라.

```text
Reference: <Jitter 템플릿 이름 + 원문 URL>
Purpose: <사용자가 무엇을 이해하거나 느껴야 하는가>
Sequence: <상태 A -> 전환 -> 상태 B>
Mechanics: <transform/opacity/mask/path/layout 등>
Adaptation: <프로젝트 토큰·컴포넌트·기술로 바꾼 점>
Reduced motion: <축소 또는 정지 대안>
Verification: <확인한 상태·뷰포트·성능>
```

레퍼런스 설명만 요청받았으면 구현하지 말고 이 브리프까지만 제공하라. 구현 요청이면 브리프를 코드와 검증으로 끝내라.
