# 구현 순서 정책 — 백엔드 우선

## 풀스택 기능 구현 순서

풀스택 기능은 **백엔드를 완성한 뒤** 프론트엔드에 착수한다. 백엔드 구간(4계층 순서):

| 단계 | 계층 | 작업 | 산출 |
|---|---|---|---|
| 1 | domain | 엔티티 → DTO(record) → 서비스 spec → 스토어 인터페이스 | `{project}-be-domain/` |
| 2 | store | 매퍼 + 쿼리 | `{project}-be-store/` |
| 3 | service | 로직 클래스 (spec 구현) | `{project}-be-service/logic/` |
| 4 | rest | 컨트롤러 (`ResponseEntity<ApiResponse<T>>`) | `{project}-be-service/rest/` |
| 5 | 검증 | API 문서 UI 또는 curl 실호출 | 올바른 봉투 응답 확인 |

프론트엔드 구간(스키마 → 서비스 → 컴포넌트 → 페이지 → 테스트)은 프론트 담당 표면 — 백엔드 검증(5) 통과가 착수 조건이다.

## 검증 게이트

| 게이트 | 확인 | 실패 시 |
|---|---|---|
| G0: 근본 원인 | 버그·회귀 과업이면 증거 있는 원인 확정 (역할의 사전 진단 게이트) | **STOP** — 편집 없이 진단 결과만 반환 |
| G1: 접근 | 서비스·API·아키텍처 확정 | **STOP** — 확정 전 코딩 금지 |
| G2: 백엔드 | 엔드포인트가 올바른 봉투를 반환 | 프론트 착수 전 백엔드 수정 |
| G3: 시크릿 | 프론트엔드 코드에 키·토큰 0건 | 즉시 백엔드 설정으로 이동 |
| G4: 패턴 | 기존 코드베이스 패턴 준수 | 유사 기능 먼저 정독 |

## 착수 전 점검

- [ ] 유사 기존 기능을 코드베이스에서 검색했다
- [ ] 아키텍처 확인 (프론트 → 백엔드 → 외부 단방향)
- [ ] 변경 필요 파일을 식별했다
- [ ] 프론트엔드 코드에 시크릿 없음
- [ ] 외부 서비스·API 를 확정했다 (`INTEGRATION.md` 결정 게이트)

## 구현 후 lint (MANDATORY)

코드 변경 후 해당 영역의 프로젝트 lint 러너를 실행한다. **lint 통과 전 완료 선언 금지** — 실패 시 자동 수정 러너 먼저, 안 되면 수동 수정.
