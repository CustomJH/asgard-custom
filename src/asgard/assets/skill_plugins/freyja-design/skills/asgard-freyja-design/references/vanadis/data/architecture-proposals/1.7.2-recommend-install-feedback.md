## Revision note (v2)

This revision corrects three load-bearing errors found in adversarial review and re-scopes the release:

1. **Pillar A decision was backwards ‚Äî flipped A2 ‚Üí A1.** The chosen A2 (lazy-resolve the reference from the
   installed package tree via `require.resolve`) is *broken for the documented primary audience*. Official
   onboarding is bare `npx vanadis-cli install-skills` (not `npm i`), so `vanadis-cli` is **never**
   written to the project's `node_modules` ‚Äî it runs from the transient npx cache that is GC'd after the run.
   `require.resolve` then throws `MODULE_NOT_FOUND`, the resolver falls through to the absent dev path, exits
   1, and the bug is **not** fixed. The only moment the reference `DESIGN.md` is reachable is *during* the
   install run, so install-skills must **copy** it into the project. We now choose **A1 (copy the chosen
   reference's `DESIGN.md` at install time)**. The inflated "18 MB" size claim is corrected to **markdown
   only, ~6 MB** (assets are not copied), and a `--no-references` lean flag is added. The resolver script is
   kept as a fallback chain whose *primary* hit is now the install-copied path.
2. **Pillar C scoring kept multiplicative.** The proposed additive `w_freq*count + w_rec*Œ£exp + w_rel*rel`
   (with `w_rel`=0.5) silently changed behavior and made AC-C6 unsatisfiable. We keep the existing
   **multiplicative** `count √ó importanceAvg √ó recency √ó 10` core, add the relevance term only as an optional
   factor gated behind `w_rel` that **defaults to 0**, and explicitly preserve the **`importanceAvg ‚â• 2.5`
   floor** (synced to `src/core/memory.ts`).
3. **Over-engineered parts deferred to 1.8.0.** Pillar B (need-detector ‚Äî unvalidated thresholds + a
   mis-grouped precedence bug in the guard), the `w_rel` relevance *weighting*, the Self-Consistency
   vote-gate, and the correction-rate reinforcement all move to 1.8.0. The cheap `VANADIS_TRIGGER=task-complete`
   trigger stays in 1.7.2 (it fills the real task-context gap). The archive-prune pass moves out of
   `vanadis-learn` Phase 5 ("Í≤∞Í≥º ÏöîÏïΩ", a semantic mismatch) into a dedicated **Phase 6 "prune."**

**Net 1.7.2 scope:** Pillar A (A1) + the deterministic, human-gated consolidation core of Pillar C
(task-complete trigger + conflict-resolution + prune; multiplicative scoring with `w_rel`=0).
**1.8.0:** Pillar B + `w_rel` weighting + vote-gate + correction-rate reinforcement.

## Executive summary

This release does two things, in priority order:

1. **Pillar A (BUG, must ship):** The `recommend ‚Üí approve ‚Üí install` chain is broken for npx/installed
   users. `vanadis-init` Phase 4.1 reads the chosen reference `DESIGN.md` from one of three paths, but
   `install-skills.ts` never lands it at path 1 (`.claude/data/references/<id>/DESIGN.md`), and paths 2‚Äì3
   only resolve in the vanadis dev repo or a vanilla npm layout. Because the documented onboarding is bare
   `npx vanadis-cli install-skills` (no `node_modules`), lazy package-tree resolution cannot work ‚Äî the
   package tree vanishes after the npx run. We **close this with option A1 (copy the chosen reference's
   `DESIGN.md` into `.claude/data/references/<id>/DESIGN.md` at install time)** plus a **fallback resolver**
   whose primary hit is that copied path. Install weight: **~6 MB of markdown** if all references are copied
   for offline browsing, or only the consumed reference per project; a `--no-references` flag opts out.
2. **Pillar C (the core):** Make DESIGN.md **measurably strengthen with use** by layering a
   recurrence-scored promotion loop on the existing `remember ‚Üí learn ‚Üí foldin` chain, with an explicit
   **conflict-resolution** step. Scoring stays the existing **multiplicative** ACT-R-style
   `count √ó importanceAvg √ó recency √ó 10` with its `importanceAvg ‚â• 2.5` floor; an optional relevance factor
   is wired behind `w_rel` (default 0, inert in 1.7.2). Grounded only in *verified* research (Reflexion,
   Generative-Agents memory-stream/reflection, Constitutional-AI critique‚Üírevise, ACT-R recency√ófrequency).
   The unverified Hermes marketing numbers are **not** cited as fact.

**Deferred to 1.8.0:** the **DESIGN.md-need detector (Pillar B)**, the `w_rel` relevance *weighting* term,
the Self-Consistency vote-gate, and the correction-rate reinforcement feedback signal ‚Äî each needs real
usage data to tune, or carries an unvalidated/mis-specified heuristic. See ¬ß6.4.

**Verdict on post-task consolidation (¬ß5):** **YES, but minimal** ‚Äî do *not* add a new hook or a new
`vanadis:consolidate` skill. Extend the existing `session-end-foldin.cjs` to *also* fire on an explicit
`epic-complete` / `task-complete` signal written to `.vanadis/timeline.md`, and have `vanadis-apply` emit that
signal when a design task closes. Reuses 100% of existing fold-in machinery.

## Pillar A ‚Äî Close the install gap (the only confirmed hard bug)

### A.0 What's actually broken

`vanadis-init` Phase 4.1 (`skills/vanadis-init/SKILL.md:87-97`) resolves the chosen reference's `DESIGN.md` via:

```
1. .claude/data/references/<id>/DESIGN.md          ‚Üê never created by install-skills
2. node_modules/vanadis-cli/web/references/<id>/DESIGN.md   ‚Üê exists in tarball, but path is brittle
3. web/references/<id>/DESIGN.md                    ‚Üê only in the vanadis dev repo
```

- **Path 1** is never populated: `install-skills.ts` `installDataFile()` (`:421-459`) only copies the 5
  flat `data/*.json` files (`dataFiles` array `:753-759`); there is no code that creates
  `.claude/data/references/<id>/`.
- **Path 2** exists in the published tarball (`package.json files[]` ships `web/references/*/DESIGN.md`),
  but the hardcoded `node_modules/vanadis-cli/...` string fails for pnpm/yarn-PnP nested layouts,
  monorepo hoisting, global installs, git installs, and bundled agents.
- **Path 3** only resolves when cwd is the vanadis source repo.

Result: for the *primary* audience (npx-installed projects), the recommend flow succeeds through Phase 3,
then Phase 4.1 fails to read any `DESIGN.md` and Phase 5 generates from nothing.

### A.1 Decision: A1 (copy reference DESIGN.md at install time) over A2 (lazy-resolve from package tree)

| | A1 ‚Äî copy `web/references/*/DESIGN.md` into `.claude/data/references/` at install | A2 ‚Äî at `vanadis:init`, resolve the chosen `<id>` from the installed package tree |
|---|---|---|
| Works for **bare `npx install-skills`** (the documented primary path) | **yes** ‚Äî copied during the install run, before the npx cache is GC'd | **no** ‚Äî `require.resolve('vanadis-cli/...')` throws `MODULE_NOT_FOUND`; no `node_modules` exists |
| Install size added | **markdown only, ~6 MB** (221 √ó ~27 KB DESIGN.md; **no assets copied**), or just the consumed reference | 0 bytes |
| Files written per project | 221 markdown files (or 1 with the lean future option) | 0 at install; 1 read at init |
| Works offline after install | yes | only if the package survives in `node_modules` (npx flow: it does not) |
| Moving parts | low (one `cpSync` of `*/DESIGN.md`) | medium (package-root resolution that cannot resolve post-npx) |
| Drift / staleness | re-run `vanadis install-skills --force` (same as skills) | n/a (but unusable for npx) |

**Chosen: A1.** The deciding factor is correctness for the *documented primary audience*. Official onboarding
is bare `npx vanadis-cli install-skills` ‚Äî it never runs `npm i`, so `vanadis-cli` is never written
to the project's `node_modules`; it executes from the transient npx cache that is garbage-collected after the
run. Under A2, `require.resolve('vanadis-cli/package.json', { paths:[projectDir] })` therefore throws
`MODULE_NOT_FOUND`, the resolver falls through to the absent dev path, and exits 1 ‚Üí the bug is **not** fixed.
(Stamping the package-root path into a json file fails for the same reason: for pure-npx that path *is* the
transient cache, which is unstable and GC'd.) The only moment the reference `DESIGN.md` files are reachable is
**during the install run**, so install-skills must **copy** them into the project then.

A1 is the only option that is both **offline-safe** *and* **npx-safe**. For a design tool whose references
*are* the product, **~6 MB of local markdown** (DESIGN.md only ‚Äî assets are not copied) is acceptable; the
earlier "18 MB" figure was inflated and assumed assets. Drift is handled exactly like skills: re-run
`vanadis install-skills --force`.

- **Lean opt-out (1.7.2):** add a `--no-references` install flag. Default = copy references; with the flag,
  no reference markdown is written and the skill surfaces the recovery message if one is later needed.
- **Future lean default (post-1.7.2, not now):** a lazy `npx vanadis-cli add-reference <id>` that fetches
  a single reference on demand. We do **not** make this the 1.7.2 default ‚Äî it reintroduces a network/registry
  dependency the offline path must not require.

> We explicitly do **not** fetch from the npm *registry* over the network in 1.7.2. The reference markdown is
> copied from the package tree *during the install run*, so the flow stays offline-safe and CI-safe.

### A.2 Concrete change 1 ‚Äî a fallback resolver (new, shared)

The resolver is still the right Phase 4.1 path resolver, but its role changes under A1: it is now a
**fallback chain whose PRIMARY hit is the install-copied path** (`.claude/data/references/<id>/DESIGN.md`),
which ‚Äî because install-skills now copies it (A.3) ‚Äî **actually exists** for the npx audience. Candidate-2
(`require.resolve`) stays as a *secondary* fallback for the `npm i` dependency layout (and pnpm/yarn/global),
and candidate-3 covers the dev repo. The runtime resolution happens inside the skill (skills have no TS
runtime), so we ship it as a small cjs helper the skill invokes.

**New file:** `scripts/vanadis-resolve-ref.cjs` (add to `package.json files[]` and to the hook/script install
list at `install-skills.ts:774-785`).

```js
#!/usr/bin/env node
// Resolve a reference DESIGN.md path for a given <id>, regardless of install layout.
// Usage: node vanadis-resolve-ref.cjs <id>  ‚Üí prints absolute path or exits 1.
'use strict';
const fs = require('node:fs');
const path = require('node:path');
const id = process.argv[2];
if (!id || !/^[a-z0-9._-]+$/i.test(id)) { process.exit(2); } // path-traversal guard
const projectDir = process.env.CLAUDE_PROJECT_DIR || process.cwd();

const candidates = [];
// 1. PRIMARY: project-local copy written by install-skills (A1). The hit for the npx audience.
candidates.push(path.join(projectDir, '.claude', 'data', 'references', id, 'DESIGN.md'));
// 2. SECONDARY fallback: resolve the installed package root via Node's own resolver
//    (only works under `npm i`/pnpm/yarn/global ‚Äî NOT under bare npx, where there is no node_modules)
try {
  const pkgJson = require.resolve('vanadis-cli/package.json', { paths: [projectDir] });
  candidates.push(path.join(path.dirname(pkgJson), 'web', 'references', id, 'DESIGN.md'));
} catch { /* not resolvable as a dep ‚Äî fall through */ }
// 3. dev repo
candidates.push(path.join(projectDir, 'web', 'references', id, 'DESIGN.md'));

for (const c of candidates) {
  if (fs.existsSync(c)) { process.stdout.write(c); process.exit(0); }
}
process.stderr.write(`vanadis: reference '${id}' DESIGN.md not found in any known location\n`);
process.exit(1);
```

Why the fallback is robust where it applies: `require.resolve('vanadis-cli/package.json', { paths:
[projectDir] })` delegates to Node's module resolution, which correctly handles pnpm symlink farms, yarn
hoisting, monorepo workspaces, and `npm root -g` global installs ‚Äî replacing a hardcoded
`node_modules/vanadis-cli` string. The `paths: [projectDir]` anchor makes it resolve relative to the
user's project. **But it only fires when the package is actually installed as a dependency** ‚Äî under bare
`npx install-skills` there is no `node_modules`, so candidate-2 throws and candidate-1 (the copied path) is
what carries the npx flow. This is exactly why A1's install-time copy is required, not optional.

## Pillar B ‚Äî DESIGN.md-need detector ‚Äî **DEFERRED TO 1.8.0**

## Pillar C ‚Äî Self-reinforcing feedback loop

### C.0 The architecture (memory-stream ‚áÑ reflection layer)

Adopt the **Generative-Agents** split directly:

- `\.vanadis/preferences.md` = **memory stream** ‚Äî append-only raw observations (already specified by
  `vanadis-remember`: each entry tagged `id / ts / scope / status:pending / importance / signal`).
- `DESIGN.md` = **reflection layer** ‚Äî the low-churn, human-readable, auditable spec. Mutated only by
  curated promotion (`vanadis-learn` / fold-in), never by raw appends. This *is* the "constitution" in CAI terms.

The loop: `vanadis-remember` writes raw ‚Üí `session-end-foldin` scores recurrence ‚Üí proposals surfaced ‚Üí `vanadis-learn`
promotes as small curated deltas ‚Üí DESIGN.md strengthens ‚Üí next run's critique pass uses the stronger spec.

### C.1 (i) Recurrence scoring & promotion thresholds

**Keep the existing multiplicative core unchanged.** `session-end-foldin.cjs:59-72` already implements an
ACT-R-style **multiplicative** activation:

```
score(scope) = count √ó importanceAvg √ó recency √ó 10
```

guarded by **three gates** (all preserved, none changed):

1. `entries.length ‚â• 3` (frequency floor),
2. **`importanceAvg ‚â• 2.5`** ‚Äî the importance floor, **explicitly synced to `src/core/memory.ts`** (the
   line ~64 comment in the hook references this duplication; it must stay in sync), and
3. `score ‚â• fold_in_score_threshold` (60),

with a hardcoded `tau = 7` (`recurrence_window_days`) in the recency factor. The earlier draft of this
section proposed replacing the multiplicative score with an **additive** `w_freq*count + w_rec*Œ£exp +
w_rel*relevance` (with `w_rel`=0.5). That silently changed behavior and made AC-C6 ("identical to 1.7.1")
unsatisfiable. **We do not do that.**

The relevance term is added only as an **optional multiplicative factor gated behind `w_rel` that DEFAULTS
TO 0** (i.e. inert), so 1.7.1 behavior is reproduced *exactly* by default:

```
relevance_factor = 1 + w_rel * scope_match_to_active_design_scope   // w_rel default 0 ‚Üí factor == 1 ‚Üí no-op
score(scope)     = count √ó importanceAvg √ó recency √ó 10 √ó relevance_factor
```

- **Promotion gate (pending ‚Üí applied):** unchanged ‚Äî `entries.length ‚â• 3` **AND** `importanceAvg ‚â• 2.5`
  **AND** `score ‚â• 60`. (The Self-Consistency vote-gate proposed earlier is **deferred to 1.8.0** ‚Äî see
  ¬ß6.4; promotion is already human-confirmed in `vanadis-learn` and again in the C.3 conflict prompt.)
- **Decay / pruning:** entries older than `recurrence_window_days` (7) already drop out of scoring; the prune
  pass that flips never-promoted, expired `pending` entries to `status:archived` is **moved to a dedicated
  `vanadis-learn` Phase 6 "prune"** (out of Phase 5 "Í≤∞Í≥º ÏöîÏïΩ", which was a semantic mismatch) ‚Äî see ¬ßC.3a. This
  keeps `preferences.md` bounded.

`w_rel` lives in `.vanadis/config.json` (the file the hook already reads) and **defaults to 0**; `w_freq`/`w_rec`
are not introduced (the score stays multiplicative, not a weighted sum). With `w_rel` absent or 0, the
relevance factor is exactly 1 and scoring is byte-identical to 1.7.1. **`w_rel` weighting stays inert until
1.8.0**, where it is tuned against real usage data (¬ß6.4).

### C.2 (ii) Consolidation trigger ‚Äî session-end exists; do we need task/epic-complete?

**Yes ‚Äî a task/epic-complete consolidation is worth adding, but as a fired event, not a new mechanism.**
Rationale: session-end is *temporal* (you closed your laptop), but design preferences cluster by *task*
("I just finished the checkout flow"). A purely session-bound fold-in mis-scores because it ignores whether
the recurring signals came from the *same* design task (the documented gap: "Missing: Task context in
fold-in scoring"). Closing a task is the natural reflection boundary (Reflexion: reflect at the end of a
trajectory, not on a wall-clock timer).

**Design (minimal):** `vanadis-apply`, when it detects a design task closed (user says "done with X" / the
harness reports a run completed), appends a marker block to `.vanadis/timeline.md`:

```
## <iso-ts> ‚Äî task_complete
scope_hint: components.button, color
```

`session-end-foldin.cjs` is extended to **also run its fold-in pass immediately when invoked with
`VANADIS_TRIGGER=task-complete`** (we add a tiny launcher: `vanadis-apply` runs
`VANADIS_TRIGGER=task-complete node .../session-end-foldin.cjs` after writing the marker). The value of this
trigger is the **timing boundary**: preferences from the just-finished task get consolidated *while they're
fresh and coherent*, instead of waiting for session end and being averaged against unrelated scopes ‚Äî and
this works with the unchanged multiplicative scoring (no weighting change needed). The `scope_hint` is
recorded so that, once `w_rel` weighting lands in **1.8.0**, the relevance factor can be boosted for those
scopes; in 1.7.2 `w_rel`=0, so the trigger purely changes *when* the existing scoring runs, not *how* it
scores. **No new hook, no new skill** ‚Äî one env-var branch in an existing hook + one Bash call from an
existing skill.

### C.3 (iii) Conflict resolution (new preference contradicts DESIGN.md)

This is the **Constitutional-AI critique‚Üírevise** step, currently missing. Add to `vanadis-learn` Phase 3
(`skills/vanadis-learn/SKILL.md:37-51`), per scope, before editing DESIGN.md:

1. **Critique:** read the current DESIGN.md rule for that scope. Does the pending preference *contradict*
   it (e.g. DESIGN.md ¬ß2 says "primary = navy"; preference says "make primary green")?
2. **Classify the conflict:**
   - *Refinement* (narrows/extends, no contradiction) ‚Üí apply as a small delta.
   - *Direct contradiction* ‚Üí **do not silently overwrite.** Surface to the user:
     *"DESIGN.md ¬ß2 says primary=navy, ÌïòÏßÄÎßå ÏµúÍ∑º 3Ìöå green ÏùÑ ÏÑÝÌò∏ÌïòÏÖ®Ïñ¥Ïöî. (a) DESIGN.md Î•º green ÏúºÎ°ú
     Î∞îÍøÄÍπåÏöî, (b) Ïù¥Î≤àÎßå ÏòàÏô∏Î°ú ÎëòÍπåÏöî, (c) Î¨¥ÏãúÌïÝÍπåÏöî?"* ‚Äî record the decision back into the preference
     entry (`status: applied | superseded | rejected`).
   - *Stale rule* ‚Üí flag for **pruning** from DESIGN.md (CAI: rules that never bind get pruned). **"Never
     binds" needs an operational definition before this fires automatically; in 1.7.2 it is a 1.8.0

This keeps DESIGN.md stable and auditable: contradictions become explicit user decisions, not silent drift.

### C.3a (i, cont.) Prune pass ‚Äî `vanadis-learn` Phase 6 (not Phase 5)

The archive-prune pass is a **new dedicated `vanadis-learn` Phase 6 "prune"** (the earlier draft put it in
Phase 5 "Í≤∞Í≥º ÏöîÏïΩ" / results summary ‚Äî a semantic mismatch). Phase 6 flips never-promoted, expired `pending`
entries (older than `recurrence_window_days`) to `status:archived`, bounding `preferences.md` (power-law
forgetting, made concrete). It runs after the Phase 5 summary so the summary reflects the pre-prune state.
(Alternatively it can run inside `session-end-foldin` after scoring; we choose the explicit `vanadis-learn`
Phase 6 so the prune is auditable and human-visible in the learn flow.)

### C.4 (iv) Feedback signal ‚Äî did the fold-in actually improve things? ‚Äî **DEFERRED TO 1.8.0**

> **Status: deferred out of 1.7.2** (consistent with ¬ß6.4). The correction-rate reinforcement needs
> accumulated `user-correction` history to be meaningful and benefits from real usage data before tuning the
> importance-prior bumps. In 1.7.2 only the **timeline reflection-line FORMAT is stubbed and
> forward-compatible** (so 1.8.0 only *reads* it); no reinforcement is computed. The full design is retained
> below as the 1.8.0 spec.

The reinforcement signal (Reflexion's verbal eval signal = user edits/corrections; RLAIF/CAI's AI-feedback
reward proxy):

> **A fold-in into scope S improved things if the rate of subsequent user *corrections* in scope S drops
> after the fold-in.**

Concrete instrumentation (data already exists ‚Äî `vanadis-remember` tags each entry `signal:user-statement` vs
`signal:user-correction`, scope-tagged, timestamped):

- On each fold-in, `vanadis-learn` records in `.vanadis/timeline.md`:
  `foldin scope=S at=T applied_pref_ids=[...]`.
- A new **reflection line** computed at the *next* task-complete/session-end for scope S:
  `correction_rate_after = (#user-correction in S since T) / (#runs touching S since T)`
  compared to `correction_rate_before`.
  - **Decreased** ‚Üí positive reinforcement: bump the *importance prior* for that scope (future signals in S
    score slightly higher / promote faster). Emit a one-line **verbal reflection** into `preferences.md`:
    *"hero copy ÏßßÍ≤å + spacing Ï¢ÅÍ≤å ÏÑÝÌò∏ ‚Üí Ïù¥ÌõÑ ÏàòÏÝï 0Í±¥. denser layout Í∑úÏπô Í∞ïÌôî."* (Reflexion verbal buffer;
    top-K relevant reflections get prepended to the next run's context per C.5).
  - **Increased/unchanged** ‚Üí negative: the rule is wrong or under-specified. Re-surface the scope in the
    next `vanadis-learn` as *"Ïù¥ Í∑úÏπô ÏÝÅÏö© ÌõÑÏóêÎèÑ ÏàòÏÝïÏù¥ Í≥ÑÏÜçÎèºÏöî ‚Äî Îã§Ïãú Î≥ºÍπåÏöî?"* (the CAI "rule keeps getting
    violated ‚áí revise or prune" path).

This makes "self-reinforcing" measurable and grounded: recurring, *validated* choices compound their weight;
rules that don't reduce corrections get revised or pruned. **No model retraining** ‚Äî the improving "policy"
is `DESIGN.md` + the reflection buffer, exactly the verified Reflexion/memory-stream mechanism.

> **Provenance note:** mechanisms above are cited from verified sources ‚Äî Reflexion (verbal self-reflection
> buffer), Generative Agents (memory-stream ‚Üí reflection promotion; recency√ófrequency√órelevance retrieval),
> Constitutional AI / RLAIF (critique‚Üírevise against a fixed rule-set), Self-Consistency (vote-gated
> promotion ‚Äî **applied in 1.8.0, not 1.7.2**), ACT-R (recency√ófrequency activation). The Hermes
> "15‚Äì30% after 5 cycles" figure and the
> "observe-reflect-optimize" three-stage optimizer are **secondary-blog / marketing claims and are NOT used
> as evidence here.**

## ¬ß5 Verdict ‚Äî post-task preference consolidation

**YES.** A task/epic-complete consolidation step *is* warranted ‚Äî but the minimal addition is **not** a new
hook and **not** a new `vanadis:consolidate` skill.

**Chosen minimal addition:** extend `session-end-foldin.cjs` to fire its existing fold-in pass on an
explicit `VANADIS_TRIGGER=task-complete` invocation, driven by a `## ... ‚Äî task_complete` marker that `vanadis-apply`
writes to `.vanadis/timeline.md` when a design task closes (¬ßC.2). Rationale:

- A new **hook** can't help: Claude Code has no native "task/epic complete" event; the only real signals are
  *session-end* (already a Stop hook) and *the agent/user declaring a task done* ‚Äî which a skill (`vanadis-apply`)
  observes far better than any hook matcher.
- A new **skill** (`vanadis:consolidate`) would duplicate `vanadis-learn` + fold-in (the brief forbids duplicating
  the existing loop).
- Extending `session-end-foldin.cjs` with one env-var branch reuses the entire scoring/proposal pipeline and
  only adds the *trigger boundary* and a relevance boost ‚Äî which is exactly the missing capability
  (task-scoped consolidation, fresh and coherent), at the cost of ~15 lines.
