# COUPLING — 결합도·응집도·변경 파급 감사

결합도는 취향이 아니라 측정값이다 — "복잡해 보인다"가 아니라 fan-in/fan-out 숫자와 변경 이력으로 판정한다.

## 측정

1. **fan-out (원심)** — 모듈이 참조하는 외부 모듈 수. import 그래프(LAYERING 절차 2)에서 집계한다. 상위 이상치가 후보다 — 절대 문턱보다 저장소 내 분포의 꼬리를 본다.
2. **fan-in (구심)** — 모듈을 참조하는 모듈 수. fan-in 높음 자체는 결함이 아니다(토대 모듈의 정상 형태) — **fan-in 과 fan-out 이 동시에 높은 모듈이 god module** 후보다.
3. **변경 동조** — `git log --format=%H --name-only` 로 최근 커밋들의 파일 집합을 뽑아, 항상 같이 바뀌는 파일 쌍(동조율 높음)이 서로 다른 모듈에 있으면 **산탄 수정**(한 변경이 여러 모듈을 관통) 신호다. 한 파일이 서로 무관한 이유로 계속 바뀌면 **발산 변경**(응집 부족) 신호다.
4. **안정 의존 원칙(SDP)** — 불안정(자주 바뀌는) 모듈에 안정 모듈이 의존하면 역전이다: 변경 빈도(git 커밋 수)를 모듈별로 집계해 `안정 → 불안정` 간선을 나열한다.

## 판정 기준

- god module(고 fan-in × 고 fan-out × 대형) = `부채` — 분해 방향(어느 책임을 어느 계층으로 내릴지) 1줄과 함께 기록.
- 산탄 수정 동조쌍 = `부채` — 관통되는 개념(응집체)을 지목한다: 같이 바뀌는 것은 같이 있어야 한다.
- SDP 역전 간선 = `부채`(인터페이스 역전 제안 동반). 단발 동조·소형 유틸 fan-in 은 `관찰`.
- 수치는 단독 FAIL 사유가 아니다 — 재현 가능한 변경 파급(실제 커밋 이력·깨진 사용처)과 결합해야 `구조 결함`으로 승격한다.

## 봉인

- god module 확정 시: 모듈 크기·의존 수 상한을 lint(예: import 수 제한 규칙) 또는 계층 테스트의 예외 목록 축소로 봉인.
- 산탄 수정 확정 시: 응집체를 한 모듈로 모은 뒤, 그 경계를 LAYERING 계층 테스트에 편입해 재발을 막는다.
