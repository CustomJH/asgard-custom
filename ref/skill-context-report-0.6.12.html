01

네 축 근거 — 원 영상 + 해설 영상

두 출처를 근거로 삼는다. 원 발표는 스킬 작성의 4단계 체크리스트를, 한국어 해설 영상은 그 문제의식(“skill hell” — tutorial hell · framework hell에 이은 세 번째: 좋은/나쁜 스킬을 가르는 공유 rubric의 부재)을 재진술한다. 각 축을 (a) 발표 원문 근거 · (b) 해설 재진술 · (c) Asgard 적용으로 1:1 정렬한다.

Trigger 트리거 · 호출 주체

발표 원문
호출 주체 결정이 내용보다 먼저. user-invoked(사람이 슬래시·수동) vs model-invoked(에이전트가 description 상시 감시 → 자동 호출). model 호출의 숨은 비용 = description이 매 요청 컨텍스트에 상주(스킬 100개면 설명 100개 부유) = context load + unpredictability(설명이 완벽해도 미호출 가능 → eval 필요). description = “context pointer”, user-invocable = 그 포인터를 에이전트에게 숨김(disable-model-invocation: true). 예: code base design=model, grill me=disable true.
해설 재진술
Pocock은 대부분 user-invoked, Superpowers는 대부분 model-invoked — 옳고 그름이 아니라 무엇을 포기할지의 trade-off.
Asgard 적용
invocation: "model"|"user" 필드로 스킬을 분류하고, user-invoked는 암묵(모델) 컨텍스트에서 제외. → §3

Structure 구조 · 절차+참고

발표 원문
스킬 내부는 정확히 두 종류: 절차(에이전트가 밟을 순서) + 참고자료. 규칙 = 본문을 최대한 작게. 특정 분기에서만 필요한 참고는 별도 파일로 빼고 본문엔 “이 템플릿 필요하면 이 파일 봐라” 한 줄 포인터만 → 필요할 때만 읽혀 평소 토큰 미소비.
해설 재진술
상시 쓰는 자료는 본문, 가끔 쓰는 자료는 말미/외부 파일로.
Asgard 적용
asgard-skillcraft: SKILL.md 본문은 5스텝 절차, 상세 rubric·deletion-test는 CHECKLIST.md 리소스로 분리. → §3

Steering 유도 · 어휘

발표 원문
가장 흔한 좌절: 분명히 썼는데 에이전트가 안 함. 진단 = 문장이 부족한 게 아니라 어휘가 약하다. 사전학습된 leading word(red-green, vertical slice, tight loop, fog of war) 하나가 재진술 문단을 이긴다. 원하는 행동을 긍정형으로.
해설 재진술
미래 단계가 보여 조기 완료가 일어나면, 먼저 현재 완료 기준을 날카롭게 — 그래도 남으면 분리.
Asgard 적용
Freyja 카탈로그의 4개 description을 job+분기만 남긴 picker 라벨로 압축(동의어=중복). → §3

Pruning 가지치기 · deletion test

발표 원문
동작하는 스킬을 얻은 뒤 최대한 작게: no-op 제거. 단 문장이 “일반적으로 들린다”는 이유로 no-op이라 단정 금지.
해설 재진술
deletion test — 문장 하나를 빼고 동일 모델·설정에서 관측 가능한 프로세스가 동등한지 확인하는 행동 증거로만 삭제.
Asgard 적용
deletion-test 프로토콜을 CHECKLIST.md에 문서화하되 CI 자동 게이트로는 편입하지 않음(§5). Canon 6도 같은 사상으로 교정(§3).
02

이미 갖춘 progressive disclosure + 바꾸지 않은 것

네 축 중 Structure의 핵심인 지연 로드는 Asgard가 이미 구현하고 있었다. 이번 릴리스는 그 위에 Trigger 격리를 얹었을 뿐, 기존 라우팅·로더·어댑터 구조는 유지한다.

이미 갖춘 것 — progressive disclosure

역할 프롬프트의 <available_skills>에는 이름 + description만 노출되고, 본문은 load_skill이 필요 시 중앙 정본에서 지연 로드한다. 모든 스킬을 미리 읽지 않는다 — 발표의 “본문 최소 + 필요 시 로드” 원칙과 이미 정합.

이번에 바꾸지 않은 것

05

채택하지 않은 과설계와 이유

발표의 “옳고 그름이 아니라 trade-off” 원칙에 따라, 단순해 보이는 강제안들을 검토 후 기각한 근거.

  • ✕모든 스킬을 무조건 user-invoked로 강제(모델 호출 폐지)
    옳고 그름 아닌 trade-off — 자율 디스커버리·스킬 간 재사용이 필요한 스킬(예: worker 디버깅/테스트)은 model-invoked가 정당. Asgard는 축별 판단을 유지한다.
  • ✕deletion-test를 CI 자동 게이트로 편입
    deletion test는 동일 모델·설정에서 관측 프로세스를 비교하는 행동 실험 — 결정론 단위테스트로 대체 불가, 상시 CI 비용 과다. CHECKLIST.md 절차로만 문서화한다.
  • ✕description을 강제 글자수 상한으로 린트
    “작을수록 결과”는 트리거 회상·프로세스 체크가 통과할 때만 성립(CHECKLIST 5.5). 맹목적 상한은 트리거 분기 손실 위험.
  • ✕Codex 전용 별도 스킬 포맷 신설
    표준 disable-model-invocation에서 파생한 agents/openai.yaml 정책 파일 하나로 최소 대응. Cursor는 무시하고 동일 디렉터리를 공유한다.
